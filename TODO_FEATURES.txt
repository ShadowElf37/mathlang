1. [PARTIAL] bytecode VM — lazy OnceLock compilation, stack-based interpreter, tensor
   indexing (Index), nested lambdas (MakeClosure), non-recursive Def::Func in blocks.

   Still possible to add to the bytecode VM:
   1a. Slices in compiled bodies: T[lo..hi] currently falls back; add a Slice instruction
   1b. Ranges in compiled bodies: lo..hi currently falls back; produces a 1-D tensor
   1c. Tensor literals: (1,2; 3,4) currently falls back; straightforward to compile
   1d. Recursive local functions: needs MakeRecFn or a Cell-based two-pass trick
   1e. [DONE v0.21.1] Loop instruction: bounded iteration (sum/prod/iterate/scan)
       compiled to a flat loop in the VM (Instruction::Loop(LoopForm, argc)) — the
       only recursion-analogue that is GPU-safe. The compiler pushes the operands
       (function + bounds/seed/count) and emits one Loop; run_vm drains them and
       calls the shared Val-based cores (agg_vals/iterate_vals/scan_vals), so
       results and errors are identical to tree-walk. These no longer force a
       fallback when used inside a compiled lambda body. (integral/deriv/map/filter/
       reduce still fall back — they need unevaluated args / bespoke handling.)
   1f. [DONE v0.21.1] Moved the Instruction enum (+ new LoopForm) to src/vm.rs; it
       depends only on crate::ast (Op, Expr), never on Val, so src/gpu/ can import
       it without pulling in eval.rs. run_vm/Compiler stay in eval.rs (they need Val).
   1g. LoadBuffer instruction: GPU-side tensor indexing for captured buffers inside
       GPU lambda bodies (v2 GPU feature, see CONSIDERATIONS.md §15)

5. [DONE v0.21.0, see docs/EVALUATION.md] Make iteration first-class, scalable, and
   collectible — the language's primary use case (time-stepping/PDEs) had no safe,
   ergonomic iteration form (recursion overflowed ~1500 deep; the scalable path was
   the undocumented `sum(k->{set(c,..);0},1,N)` cell hack). All five shipped:
   5a. [DONE] iterate(f, x0, n) — apply f n times; flat internal loop, O(1) stack,
       scales to 1e6. Safe replacement for the recursive evolve() idiom. Special form
       (eval_iterate). x0 may be scalar/vector/tuple/complex.
   5b. [DONE] scan(f, x0, n) — returns the whole orbit [x0, f(x0), ..., f^n(x0)]
       stacked into a tensor (scalar states -> 1-D [n+1]; vector states -> 2-D rows
       [n+1, d]). Special form (eval_scan) backed by the stack_rows helper. Retires
       the cell+sum trajectory hack.
   5c. [DONE] cumsum / cumprod / diff — array scans over 1-D tensors/tuples.
   5d. [DONE] Singleton collection literal: (x,) is now a length-1 tensor (parser
       trailing-comma -> Expr::Array), identical to [x]; (x) alone is still scalar x.
   5e. [DONE] Generalized stacking: vstack/hstack rank-promote (a 1-D vector becomes a
       row/column when stacked onto a matrix; scalar -> 1x1) via the to_block helper;
       append accepts a scalar seed; concat accepts scalars and empty operands.
   Follow-up: 1e (VM Loop instruction) — DONE v0.21.1 — now lets iterate/scan/sum/prod
   compile *inside* lambda bodies via Instruction::Loop instead of forcing a tree-walk
   fallback. FEAT-F is therefore complete.

6. Tame the builtin namespace. The global builtin table has ~150 names, and every
   one is a reserved word: is_protected() (src/eval.rs ~563) bars the user from
   binding any of them as a variable. Two distinct costs — discoverability (the
   flat !help list is a wall) and namespace pollution (the reserved-word set). Do
   them in two stages, lowest-risk first:

   6a. [PRIMARY] Prelude extraction. Move the "library tier" builtins — special
       functions and niche stats (erf, erfc, j0/j1, jinc, gaussian_cdf, sinc,
       maybe median/mode, the integrators once stable) — out of Rust into an
       auto-loaded mathlang prelude (.math), built on the MATHLANG_INIT / import_file
       machinery that already exists. Keep a small frozen *core* protected (~40
       names: arithmetic, tensor construction/indexing, cell/get/set, the loop
       forms iterate/scan/sum/prod, if, map/filter/reduce). Net effect: the Rust
       builtin table and is_protected both shrink dramatically, AND prelude-defined
       names become *user-overridable* (they're ordinary defs, not protected) — a
       strict ergonomics win. No grammar change; only an auto-load hook + moving
       definitions. This is the high-value/low-risk move and should land first.

   6b. [OPTIONAL, later] Additive namespace values, only if 6a leaves discoverability
       wanting. Expose grouped modules — linalg, stats, signal, bits — as first-class
       Val::Namespace map values so `linalg.det(A)` works and a namespace can be bound
       or passed around. Keep all current flat names working (back-compat); new niche
       builtins then go *only* into a namespace, capping global growth. Feasibility:
       the lexer already errors on a bare `.` not followed by a digit (src/lexer.rs
       ~88) and disambiguates decimals by digit-lookahead, so `.` is free for member
       access with no grammar conflict. Needs: a Val::Namespace variant, a parse node
       for `ident.ident` access, and resolution in eval + the VM (LoadCaptured-style).
       Do NOT move existing names *out* of the global table (that breaks every script);
       namespaces are an additive alias layer, not a replacement.

   Rationale: 6a is the honest dividend of "is the infra good enough to build in
   mathlang" — yes, so push the library tier into mathlang itself. 6b is the
   long-term shape for a browsable stdlib but shouldn't gate 6a.
